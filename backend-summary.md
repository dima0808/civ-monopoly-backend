# Серверна частина (Backend) гри "Civilization Monopoly"

## 1. Загальний опис

Серверна частина гри "Civilization Monopoly" являє собою багатокористувацький ігровий сервер, побудований на основі фреймворку Spring Boot. Додаток реалізує варіацію настільної гри "Монополія" з тематикою серії ігор Civilization, де гравці змагаються за контроль над ресурсами, будують райони, зводять чудеса світу та просуваються через історичні епохи. Серверна частина забезпечує повний ігровий цикл: від реєстрації та автентифікації користувачів до управління ігровими кімнатами, ходами гравців, економікою та спілкуванням у реальному часі.

## 3. Архітектура додатку

Серверна частина побудована за принципом **тришарової архітектури** (three-tier architecture), що забезпечує чіткий поділ відповідальностей:

### 3.1. Шар представлення (Presentation Layer)

Шар представлення відповідає за обробку HTTP-запитів та WebSocket-повідомлень. Реалізований через REST-контролери (`web/`) та конфігурацію WebSocket. Контролери приймають запити від клієнтів, валідують вхідні дані, делегують обробку сервісному шару та формують відповіді у форматі JSON.

### 3.2. Шар бізнес-логіки (Service Layer)

Сервісний шар (`service/`) містить всю бізнес-логіку гри: управління ходами, обчислення вартості нерухомості, перевірку вимог до покращень, обробку подій та управління таймерами. Сервіси реалізовані через інтерфейси та їх імплементації (`service/impl/`), що забезпечує слабку зв'язність та можливість тестування.

### 3.3. Шар доступу до даних (Data Access Layer)

Шар доступу до даних (`repository/`) реалізований через Spring Data JPA репозиторії та сутності Hibernate (`repository/entity/`). Кожна сутність відображається на відповідну таблицю в базі даних PostgreSQL. Hibernate автоматично генерує та оновлює схему бази даних на основі анотацій сутностей (`spring.jpa.hibernate.ddl-auto=update`).

### 3.4. Діаграма залежностей

```
[Клієнт] <── HTTP/WebSocket ──> [Контролери (web/)]
                                        │
                                        ▼
                              [Сервіси (service/)]
                                   │         │
                                   ▼         ▼
                          [Репозиторії]  [WebSocket Broadcast]
                               │
                               ▼
                         [PostgreSQL]
```

## 5. Модель даних

### 5.1. Сутність User (Користувач)

Представляє зареєстрованого користувача системи. Реалізує інтерфейс `UserDetails` Spring Security для інтеграції з механізмом автентифікації.

| Поле | Тип | Опис |
|---|---|---|
| `reference` | UUID | Унікальний ідентифікатор (первинний ключ, генерується автоматично) |
| `username` | String | Унікальне ім'я користувача |
| `password` | String | Пароль, зашифрований алгоритмом BCrypt |
| `avatarUrl` | String | URL аватара (за замовчуванням: `/images/default/avatar.png`) |
| `member` | Member | Зв'язок один-до-одного з поточним ігровим учасником |
| `authorities` | Set\<Authority\> | Набір ролей (ROLE_USER, ROLE_ADMIN) |

### 5.2. Сутність Room (Ігрова кімната)

Представляє ігрову кімнату, в якій проходить партія. Перший учасник автоматично стає власником кімнати.

| Поле | Тип | Опис |
|---|---|---|
| `reference` | UUID | Унікальний ідентифікатор кімнати |
| `name` | String | Назва кімнати |
| `memberLimit` | Integer | Максимальна кількість гравців |
| `password` | String | Пароль для входу (опціонально) |
| `isStarted` | Boolean | Прапорець початку гри |
| `turnIndex` | Integer | Індекс поточного гравця в черзі ходів |
| `isDiceRolled` | Boolean | Чи було кинуто кубики в поточному ході |
| `turn` | Integer | Номер поточного ігрового ходу (раунду) |
| `members` | List\<Member\> | Список учасників (відсортований за часом приєднання) |

### 5.3. Сутність Member (Учасник гри)

Представляє гравця в контексті конкретної ігрової сесії з усіма ігровими параметрами.

| Поле | Тип | Опис |
|---|---|---|
| `reference` | UUID | Унікальний ідентифікатор учасника |
| `position` | Integer | Позиція на ігровому полі (0–47) |
| `gold` | Integer | Кількість золота (ігрова валюта) |
| `strength` | Integer | Військова сила |
| `tourism` | Integer | Рівень туризму (культурний вплив) |
| `score` | Integer | Загальний рахунок гравця |
| `roundsMade` | Integer | Кількість зроблених кіл по полю |
| `civilization` | Civilization | Обрана цивілізація (10 варіантів + RANDOM) |
| `color` | Color | Колір гравця (8 варіантів) |
| `user` | User | Зв'язок з обліковим записом користувача |
| `room` | Room | Зв'язок з ігровою кімнатою |
| `properties` | List\<Property\> | Список придбаних властивостей |
| `events` | List\<Event\> | Список ігрових подій гравця |

**Доступні цивілізації:** AMERICA, AZTEC, BABYLON, COLOMBIA, EGYPT, ENGLAND, FRANCE, GAUL, JAPAN, OTTOMAN.

**Доступні кольори:** RED, BLUE, GREEN, YELLOW, TURQUOISE, ORANGE, PINK, VIOLET.

### 5.4. Сутність Property (Ігрова властивість)

Представляє придбану гравцем клітинку на ігровому полі з системою покращень та бонусів.

| Поле | Тип | Опис |
|---|---|---|
| `reference` | UUID | Унікальний ідентифікатор |
| `position` | Integer | Позиція на полі (0–47) |
| `mortgage` | Integer | Статус іпотеки (-1 = немає; 0–5 = раундів до вилучення) |
| `roundOfLastChange` | Integer | Раунд останньої зміни (для перевірки часових вимог) |
| `turnOfLastChange` | Integer | Хід останньої зміни |
| `upgrades` | List\<UpgradeType\> | Рівні покращень (LEVEL_1 – LEVEL_4_3) |
| `bonuses` | List\<BonusType\> | Активні бонуси (17 типів, наприклад COLOSSEUM, TEMPLE_OF_ARTEMIS) |
| `member` | Member | Зв'язок з власником |
| `room` | Room | Зв'язок з ігровою кімнатою |

**Рівні покращень:** LEVEL_1 (базовий), LEVEL_2, LEVEL_3, LEVEL_4, LEVEL_4_1, LEVEL_4_2, LEVEL_4_3 (спеціалізовані гілки четвертого рівня).

### 5.5. Сутність Event (Ігрова подія)

Представляє ігрову подію, що відбувається під час гри.

| Поле | Тип | Опис |
|---|---|---|
| `reference` | UUID | Унікальний ідентифікатор |
| `eventType` | EventType | Тип події |
| `extraData` | EventExtraData | Додаткові дані про подію |
| `member` | Member | Гравець, якого стосується подія |

**Типи подій:** BUY_PROPERTY (купівля), FOREIGN_PROPERTY (чужа властивість), ENEMY_PROPERTY (ворожа властивість), GOODY_HUT (знахідка), BARBARIANS (варвари), PROJECTS_* (проєкти), ALLIANCE (альянс), PEACE (мир).

### 5.6. Сутності Chat та Message (Чат та повідомлення)

Система чатів підтримує публічний (загальний для кімнати) та приватний (між двома користувачами) режими.

**Chat:**
| Поле | Тип | Опис |
|---|---|---|
| `reference` | UUID | Унікальний ідентифікатор чату |
| `messages` | List\<Message\> | Список повідомлень |
| `users` | Set\<User\> | Учасники чату |
| `mutedUsers` | List\<String\> | Заглушені користувачі |

**Message:**
| Поле | Тип | Опис |
|---|---|---|
| `reference` | UUID | Унікальний ідентифікатор повідомлення |
| `content` | String | Текст повідомлення |
| `timestamp` | LocalDateTime | Час відправки |
| `sender` | User | Автор повідомлення |

### 5.7. Сутність Authority (Роль доступу)

| Поле | Тип | Опис |
|---|---|---|
| `id` | Long | Ідентифікатор |
| `name` | AuthorityName | Роль (ROLE_USER або ROLE_ADMIN) |

## 7. Комунікація в реальному часі (WebSocket)

### 7.1. Конфігурація WebSocket

Серверна частина використовує протокол **STOMP** (Simple Text Oriented Messaging Protocol) поверх WebSocket для забезпечення двонаправленого зв'язку в реальному часі. Точка підключення: `/ws` із резервним механізмом SockJS для браузерів без підтримки WebSocket.

Конфігурація брокера повідомлень:
- `/topic` — для широкомовних повідомлень (всім підписникам)
- `/user` — для персональних повідомлень (конкретному користувачу)
- `/queue` — для черг повідомлень

### 7.2. Канали повідомлень

| Канал | Призначення |
|---|---|
| `/topic/games/{roomReference}` | Оновлення стану гри (всі гравці кімнати) |
| `/topic/chats/{chatReference}` | Повідомлення чату (публічного або приватного) |
| `/topic/rooms/{roomReference}` | Зміни у складі кімнати |
| `/user/{username}/queue/notifications` | Персональні сповіщення |

### 7.3. Типи ігрових повідомлень (GameMessage)

| Тип | Опис |
|---|---|
| `START` | Гру розпочато |
| `ROLL_DICE` | Гравець кинув кубики |
| `FORCE_ROLL_DICE` | Таймер примусово кинув кубики |
| `END_TURN` | Хід завершено |
| `FORCE_END_TURN` | Таймер примусово завершив хід |

### 7.4. Типи повідомлень кімнати (RoomMessage)

| Тип | Опис |
|---|---|
| `CREATE` | Кімнату створено |
| `JOIN` | Гравець приєднався |
| `LEAVE` | Гравець вийшов |
| `KICK` | Гравця виключено |
| `DELETE` | Кімнату видалено |
| `CHANGE_CIVILIZATION` | Цивілізацію змінено |
| `CHANGE_COLOR` | Колір змінено |

### 7.5. Автентифікація WebSocket

Автентифікація WebSocket-з'єднань здійснюється через перехоплювач каналів (`WebSocketAuthChannelInterceptor`), який:
- Перехоплює команди SUBSCRIBE
- Валідує JWT-токен з заголовка Authorization
- Перевіряє право доступу до публічних та приватних чатів
- Забезпечує, що користувач може підписатися лише на свої власні персональні канали

## 8. Ігрова логіка

### 8.1. Ігровий цикл

1. **Створення кімнати**: Власник створює кімнату із зазначенням назви, ліміту гравців та необов'язкового паролю.
2. **Підготовка**: Гравці приєднуються та обирають цивілізацію та колір. Дублікати заборонені.
3. **Старт гри**: Власник ініціює початок. Гравці з вибором RANDOM отримують випадкову цивілізацію. Перший гравець обирається випадковим чином.
4. **Хід гравця**: Поточний гравець кидає кубики (два шестигранні кубики), переміщується по полю, здійснює дії (купівля, покращення, іпотека) та завершує хід.
5. **Зміна ходу**: Черга переходить до наступного гравця. Коли всі гравці зробили хід, номер раунду збільшується.

### 8.2. Ігрове поле

Ігрове поле складається з **48 клітинок** (позиції 0–47), розташованих по колу. Кожна клітинка представляє певний тип ігрової властивості. Переміщення гравця здійснюється за формулою:

```
нова_позиція = (поточна_позиція + кубик_1 + кубик_2) % 48
```

### 8.3. Типи властивостей

**Ресурси (RESOURCE):** Базові сировинні клітинки (Коні, Банани, Олені, Хутро, Залізо, Краби, Рифи, Кукурудза, Пшениця, Рис). Мають 3 рівні покращень: базовий ресурс → покращений збір → промислова обробка.

**Райони (DISTRICT):** Спеціалізовані ігрові зони з 4 рівнями покращень:
- **Казарми** (Encampment): Казарми → Зброярня → Військова академія
- **Урядова площа** (Government Plaza): Ключова будівля з кількома рівнями
- **Промислова зона** (Industrial Zone): Майстерня → Фабрика → Вугільна електростанція
- **Кампус** (Campus): Бібліотека → Університет → Дослідницька лабораторія
- **Гавань** (Harbor): Маяк → Верф → Морський порт
- **Торговий центр** (Commercial Hub): Ринок → Банк → Фондова біржа
- **Театральна площа** (Theater Square): Амфітеатр → Художній музей → Телецентр
- **Розважальний комплекс** (Entertainment Complex): Арена → Зоопарк → Стадіон
- **Житловий район** (Neighborhood): Продовольчий ринок → Торговий центр
- **Акведук**, **Дамба**, **Космодром**

**Чудеса світу (WONDER):** Унікальні високовартісні властивості, що потребують попередньо побудованих будівель:
- Храм Артеміди (вимагає OWN_CAMP)
- Альгамбра (вимагає OWN_ENCAMPMENT)
- Велика бібліотека (вимагає OWN_LIBRARY)
- Каса де Контратасьйон (вимагає OWN_GOVERNMENT_PLAZA)
- Колізей (вимагає OWN_ARENA)
- Етеменанкі (без вимог)
- Мавзолей у Галікарнасі (вимагає OWN_HARBOR)
- Рурська долина (вимагає OWN_FACTORY)
- Стадіон Маракана (вимагає OWN_STADIUM)
- Біг Бен (вимагає OWN_STOCK_EXCHANGE)
- Оксфордський університет (вимагає OWN_UNIVERSITY)

### 8.4. Система вимог

Система вимог (`Requirement.java`) налічує понад 40 різних умов, які необхідно виконати для покращення властивостей:

**Часові вимоги:**
- MAKE_ONE_ROUND, MAKE_TWO_ROUNDS, MAKE_THREE_ROUNDS — кількість кіл по полю після придбання
- MAKE_ONE_TURN – MAKE_FOUR_TURNS — кількість ігрових ходів після придбання

**Вимоги до володінь:**
- OWN_ENCAMPMENT, OWN_HARBOR, OWN_CAMP, OWN_LIBRARY, OWN_UNIVERSITY, OWN_ARENA, OWN_STADIUM, OWN_FACTORY, OWN_STOCK_EXCHANGE, OWN_GOVERNMENT_PLAZA
- OWN_SPACEPORT_OR_LAB, OWN_TWO_WONDERS, OWN_TWO_RESOURCES
- OWN_NO_GOVERNMENT_PLAZA — взаємовиключна вимога

**Вимоги до ресурсів:**
- HAVE_LOW_GOLD (500+), HAVE_MEDIUM_GOLD (1500+), HAVE_HIGH_GOLD (2000+)
- HAVE_LOW_STRENGTH (400+), HAVE_MEDIUM_STRENGTH (800+), HAVE_HIGH_STRENGTH (1200+)
- HAVE_MEDIUM_TOURISM (300+), HAVE_HIGH_TOURISM (600+)
- HAVE_LOW_GOLD_PER_TURN, HAVE_MEDIUM_GOLD_PER_TURN, HAVE_HIGH_GOLD_PER_TURN

**Вимоги до розміру імперії:**
- TALL_EMPIRE (9+ покращень вище LEVEL_1), SUPER_TALL_EMPIRE (15+)
- WIDE_EMPIRE (7+ властивостей), SUPER_WIDE_EMPIRE (11+)

**Вимоги до ери:**
- ON_CLASSICAL_ERA, ON_MEDIEVAL_ERA, ON_RENAISSANCE_ERA, ON_INDUSTRIAL_ERA, ON_MODERN_ERA, ON_ATOMIC_ERA

### 8.5. Система ер

Гра просувається через **7 історичних епох**, що відкриваються на основі номера ходу:

| Ера | Назва | Початок (хід) |
|---|---|---|
| CLASSICAL | Класична | 10 |
| MEDIEVAL | Середньовічна | 25 |
| RENAISSANCE | Ренесанс | 40 |
| INDUSTRIAL | Індустріальна | 55 |
| MODERN | Сучасна | 70 |
| ATOMIC | Атомна | 85 |
| INFORMATION | Інформаційна | 100 |

Кожна ера розблоковує нові рівні покращень для будівель та чудес світу, створюючи поступову криву прогресу від простих будівель до передових технологій.

### 8.6. Економічна система

Кожен гравець має три ключові ресурсні показники:

- **Золото (Gold):** Основна валюта для купівлі та покращення властивостей. Накопичується через дохід від властивостей (GPT — Gold Per Turn).
- **Військова сила (Strength):** Відображає військову потужність гравця. Необхідна для розблокування військових покращень.
- **Туризм (Tourism):** Культурний вплив гравця. Розблоковує культурні будівлі та покращення.

**Система доходів (Gold Per Turn):** Кожна властивість з покращеннями генерує певну кількість золота за хід. Загальний дохід обчислюється як сума GPT всіх володінь гравця.

**Система витрат на армію:** Залежно від рівня витрат на армію, гравець отримує різне співвідношення військової сили та золота:
- Мінімальні витрати: -50 сили, +30 золота
- Помірні витрати: +10 сили, 0 золота
- Середні витрати: +100 сили, -200 золота
- Максимальні витрати: +200 сили, -700 золота

### 8.7. Система іпотеки

Гравець може заставити (іпотекувати) властивість для отримання негайних коштів:

1. **Іпотека:** Гравець отримує 40% від базової ціни властивості. Лічильник іпотеки встановлюється на 5 раундів.
2. **Відлік:** Кожен ігровий раунд лічильник зменшується на 1.
3. **Автоматичне вилучення:** Коли лічильник досягає 0, властивість автоматично видаляється.
4. **Викуп:** Гравець може викупити іпотечну властивість, сплативши 60% від початкової ціни.
5. **Зниження рівня:** Гравець може знизити рівень покращення, отримавши повне повернення вартості цього рівня.

### 8.8. Система бонусів

Властивості можуть генерувати додаткові бонуси за виконання певних умов:
- **Бонуси суміжності (тип 1):** Додатковий дохід за побудову поряд із сумісними районами.
- **Бонуси чудес (тип 0):** Спеціальні бонуси від побудованих чудес світу.

### 8.9. Система таймерів

Для підтримки динаміки гри реалізована система примусових таймерів:

- **Таймер ходу (TURN):** 60 секунд на кидок кубиків. Якщо гравець не кинув кубики за відведений час, система примусово кидає кубики (FORCE_ROLL_DICE).
- **Таймер після кидка (DICE):** 5 секунд після кидка кубиків для здійснення дій. Якщо гравець не завершив хід, система примусово завершує його (FORCE_END_TURN).

Планувальник таймерів (`GlobalGameScheduler`) працює з частотою 1 секунда, відправляючи оновлення таймера всім підключеним гравцям через WebSocket.

## 9. Система автентифікації та безпеки

### 9.1. JWT-автентифікація

Серверна частина використовує **JSON Web Tokens (JWT)** для безстанової (stateless) автентифікації:

1. **Реєстрація:** Користувач надає ім'я та пароль → пароль шифрується BCrypt → створюється обліковий запис із роллю ROLE_USER → генерується JWT-токен.
2. **Вхід:** Валідація облікових даних через AuthenticationManager → генерація JWT-токену.
3. **Запит:** Клієнт включає заголовок `Authorization: Bearer {token}` у кожен запит.
4. **Валідація:** Фільтр `HttpAuthTokenFilter` перехоплює кожен запит, валідує JWT-токен та встановлює контекст безпеки.

**Параметри токену:**
- Алгоритм підпису: HS512
- Термін дії: 24 години (86 400 000 мс)
- Корисне навантаження: ім'я користувача (subject)

### 9.2. Розмежування доступу

- **@PreAuthorize("hasRole('USER')"):** Більшість ігрових операцій доступні лише автентифікованим користувачам.
- **@PreAuthorize("hasRole('ADMIN')"):** Адміністративні функції (заглушення користувачів у чаті).
- **Публічні ендпоінти:** Ігрова конфігурація, список кімнат, перегляд профілів.

### 9.3. Конфігурація безпеки

- **CORS:** Дозволені всі джерела (для розробки).
- **CSRF:** Вимкнено (не потрібно для API з JWT).
- **Сесії:** Режим STATELESS (без збереження стану на сервері).
- **Кодування паролів:** BCryptPasswordEncoder.

## 10. Конфігурація через JSON-файли

Ігрові механіки повністю визначаються через конфігураційні JSON-файли, що дозволяє змінювати баланс гри без модифікації коду:

### 10.1. properties.json

Містить визначення **47 ігрових властивостей** із повним описом для кожної:
- Назва та тип властивості
- Рівні покращень (LEVEL_1 – LEVEL_4_3)
- Ціна кожного рівня покращення
- Вартість операції (GOS — Gold Operation Start)
- Дохід за хід (GPT — Gold Per Turn)
- Вимоги для кожного рівня покращення
- Бонуси (ефекти чудес та суміжності)

### 10.2. game.json

Визначає:
- Пороги ходів для кожної історичної ери
- Таблицю витрат на армію (співвідношення сили та золота)

### 10.3. requirements.json

Містить порогові значення для всіх ресурсних вимог:
- Золото: LOW (500), MEDIUM (1500), HIGH (2000)
- Сила: LOW (400), MEDIUM (800), HIGH (1200)
- Туризм: MEDIUM (300), HIGH (600)
- Розмір імперії: TALL (9), SUPER_TALL (15), WIDE (7), SUPER_WIDE (11)

## 11. Обробка помилок

Серверна частина реалізує систему користувацьких винятків для кожного домену:

- **Користувачі:** UserNotFoundException, UserAlreadyExistsException, UserAlreadyInRoomException
- **Кімнати:** RoomNotFoundException, RoomIsFullException, InvalidRoomPasswordException

## 12. Перетворення даних (DTO та MapStruct)

Для забезпечення чіткого розділення між внутрішніми сутностями бази даних та зовнішнім API використовуються об'єкти передачі даних (DTO). Автоматичне перетворення між сутностями та DTO здійснюється через бібліотеку MapStruct:

- `UserMapper` — перетворення User ↔ UserDto
- `RoomMapper` — перетворення Room ↔ RoomDto
- `MemberMapper` — перетворення Member ↔ MemberDto
- `PropertyMapper` — перетворення Property ↔ PropertyDto
- `ChatMapper` — перетворення Chat ↔ ChatDto
- `MessageMapper` — перетворення Message ↔ MessageDto

## 14. Висновок

Серверна частина гри "Civilization Monopoly" являє собою повноцінний багатокористувацький ігровий сервер, побудований на сучасному технологічному стеку Java/Spring Boot. Архітектура додатку забезпечує:

- **Масштабованість** — тришарова архітектура з чітким розділенням відповідальностей
- **Безпеку** — JWT-автентифікація з рольовим розмежуванням доступу
- **Реальний час** — WebSocket/STOMP для миттєвого оновлення стану гри
- **Гнучкість** — конфігураційний підхід до ігрових механік через JSON-файли
- **Розширюваність** — модульна структура з використанням інтерфейсів та патернів проєктування
- **Надійність** — система обробки помилок із користувацькими винятками для кожного домену

Проєкт демонструє застосування професійних практик розробки: ін'єкція залежностей (Dependency Injection), управління транзакціями, контроль доступу на основі ролей, валідація даних та чисте розділення бізнес-логіки від рівня представлення.
